# PrintGenerator

Welcome to the **PrintGenerator**! This application is a fully integrated Template Management System. It is built utilizing the modern MERN stack architecture (MongoDB, Express/Next.js APIs, React, Next.js) tailored for one highly robust functionality: Designing complex layout templates and batch-generating populated, beautiful PDF files compiled dynamically from JSON parameter inputs.

Users can draft rich-text template formats visually through the platform, insert programmatic placeholders like `{{name}}`, and push large datasets into the system to instantaneously receive an archived ZIP containing dozens of perfectly mapped PDF files.

## 🚀 Setup & Installation Guide

This guide is intended to be beginner-friendly. Follow this setup to get the server running locally on your machine.

### Prerequisites
Before starting, ensure you have the following installed on your system:
- **Node.js**: The Javascript runtime (Version 18+ recommended).
- **npm**: The Node Package Manager (installs alongside Node.js).
- **Git** (optional): Used for cloning the repository.

#### System Dependencies (Linux/WSL)
If you're running on **Linux or WSL**, you'll need to install system libraries required for Puppeteer/Chromium PDF generation:
```bash
sudo apt update && sudo apt install -y libnspr4 libnss3 libgconf-2-4 libatk1.0-0 libgbm1 libnss3 libxss1 libxtst6
```
For **macOS**: These are pre-installed with the OS.
For **Windows**: Chromium dependencies are typically handled automatically.

### 1. Clone & Download the Repository
If you downloaded the code as a folder, simply open it in your terminal. If you are using Git, clone the project and navigate into the root directory:
```bash
git clone <repository_url>
cd PrintGenerator
```

### 2. Install Project Dependencies
Run the install command to configure the local `/node_modules/` folder with all essential structural libraries (like React, Tiptap, Puppeteer for PDF headless chrome generation, etc).
```bash
npm install
```

### 3. Setup Environment Variables
The codebase relies on environment variables to know where to save its templates.
In the root directory, you should see a `.env.local` file. If not, create a file named `.env.local` specifically and populate it with your specific MongoDB variables:
```env
MONGODB_URI=mongodb://localhost:27017 # Or your Atlas cluster connect link
MONGODB_DB=print-generator-local
```
*(If you do not have a live MongoDB instance, the system internally detects timeouts and falls back seamlessly to an ephemeral Memory Server automatically!)*

### 4. Running the Development Server
With the database variables mapped and packages initialized, you're ready to start the application:
```bash
npm run dev
```

The application interface will bind to the browser at **http://localhost:3000**.
Navigate there, and you will see the interactive Template Layout Dashboard. You can also view the documented API Swagger specification at **http://localhost:3000/api-docs**.

## 🔧 Troubleshooting

### Chromium/Puppeteer Errors on Linux/WSL
If you encounter errors related to missing libraries when generating PDFs (e.g., `libnspr4.so` not found), run:
```bash
sudo apt update && sudo apt install -y libnspr4 libnss3 libgconf-2-4 libatk1.0-0 libgbm1 libnss3 libxss1 libxtst6
```
Then restart your development server with `npm run dev`.

### MongoDB Connection Issues
If the system cannot connect to MongoDB:
- Ensure MongoDB is running locally: `mongod` or check your Atlas connection string in `.env.local`
- The system will automatically fall back to an in-memory database if MongoDB is unavailable
- Check `.env.local` exists with valid `MONGODB_URI` and `MONGODB_DB` values

## 📂 File Structure & Code Documentation

Below is an exhaustive line-by-line guide explaining the purpose of **every single code file** found in this extensive project.

### Root Directory (Configurations)
- **`package.json`**: Manages the project's meta-information, executable shell scripts (like `dev` and `test`), and lists all external libraries needed.
- **`package-lock.json`**: Autogenerated locking file ensuring uniform library versions install exactly the same on any computer.
- **`tsconfig.json`**: Configures the TypeScript compiler detailing strictness, typing constraints, and root paths.
- **`next.config.ts`**: The highest-level Next.js configuration defining server routing, building capabilities, and global build settings.
- **`vitest.config.ts`**: The configuration orchestrating and mounting the Vitest testing framework for all automated assertions.
- **`vitest.setup.ts`**: Executed immediately prior to testing runs to bootstrap dummy variables or test environment hooks.
- **`eslint.config.mjs`**: Declares formatting style standards and programmatic warnings/errors preventing poor coding logic syntax.
- **`postcss.config.mjs`**: Configuration utilized to parse standard CSS dynamically targeting environments with external utilities.
- **`run-tests.js`**: An independent node script executing integration sequences mapping tests manually.
- **`next-env.d.ts`**: Autogenerated invisible declaration definitions connecting base module typing properties automatically.
- **`.env.local`**: Private hidden mapping parameters declaring secret environment states like database connection strings.

### The Application (`app/`)
Next.js utilizes "App Routing", translating folders inside `app/` directly into URL structures!
- **`app/page.tsx`**: The primary root dashboard page application driving visual grids, state, and interaction bindings.
- **`app/layout.tsx`**: Universal HTML wrapper determining physical styling, headers, rendering injection, and metadata titles.
- **`app/api-docs/page.tsx`**: Renders the complete frontend Swagger-UI API testing dashboard viewer targeting local routes natively.

### The Backend APIs (`app/api/`)
- **`app/api/docs/route.ts`**: Returns the massive OpenAPI JSON schema describing backend capabilities dynamically.
- **`app/api/templates/route.ts`**: Endpoint pipeline catching `GET` layouts and `POST` commands initializing brand new templates.
- **`app/api/templates/[id]/route.ts`**: Endpoint designed specifically handling mutations (`PUT`, `DELETE`) on one specific targeted template ID.
- **`app/api/templates/[id]/generate/route.ts`**: The critical powerhouse endpoint iterating JSON datapoints to orchestrate array-driven PDF conversions exported as Zips.
- **`app/api/tags/route.ts`**: Handles creation, reading, and renaming generic reusable Tags.
- **`app/api/tags/[name]/route.ts`**: Allows deletion of specific Tags while automatically nullifying associations across all layouts cleanly.
- **`app/api/tags/[name]/templates/route.ts`**: Fast resolution endpoint returning detailed templates leveraging optimized two-way database maps natively.

### React Visual Components (`app/components/`)
- **`app/components/CreateTemplateModal.tsx`**: Popup window collecting variables rendering the editor meant specifically to build a new template.
- **`app/components/EditTemplateModal.tsx`**: Popup window hydrating previously saved schemas allowing users to dynamically modify their design.
- **`app/components/GenerateModal.tsx`**: Pipeline interface consuming standard JSON to interface directly with batch endpoint PDF creations.
- **`app/components/TemplateCard.tsx`**: Displays simplified summary versions and timestamps cleanly organizing visual hierarchies into a grid loop.
- **`app/components/TemplateEditor.tsx`**: Rich-Text WYSIWYG formatting powerhouse integrating Tiptap rendering allowing programmatic target insertion.
- **`app/components/TagManagementSection.tsx`**: Renders the complete table interface summarizing all tags, usages, and providing quick access interaction buttons.
- **`app/components/CreateTagModal.tsx` & `EditTagModal.tsx`**: Dedicated simple dialog prompts capturing strictly-validated tag name configurations.
- **`app/components/DeleteConfirmModal.tsx`**: Generic warning confirmation interceptor blocking immediate destructive API actions directly.
- **`app/components/TagTemplatesModal.tsx`**: Dynamic modal re-rendering isolated grids explicitly rendering only template components referencing the currently selected Tag ID.

### Underlying Utilities (`lib/`)
- **`lib/document-generation.ts`**: The primary compilation engine deeply substituting JSON combinations through layout ASTs mounting headless Chrome environments creating PDFs.
- **`lib/mongodb.ts`**: Connects, tracks, and isolates concurrent requests protecting backend databases from crashing under heavy API calls.
- **`lib/swagger.ts`**: Statically renders and registers OpenAPI standard schemas exposing developer interactive endpoint testing interfaces natively.
- **`lib/tiptap/placeholder.ts`**: An integrated architectural module converting raw variables syntactically into stylized blocks natively visible in the browser DOM.

### Database Definitions (`types/`)
- **`types/template.ts`**: Essential centralized definitions dictating universally formatted database objects verifying strict typing.
- **`types/tag.ts`**: Establishes standard variable layouts tracking strict dual-reference models for metadata categorization.

### The Testing Suite (`__tests__/`)
- **`__tests__/api/templates/route.test.ts`**: Test environments explicitly validating collection-level endpoints like fetching grids or inserting lists successfully.
- **`__tests__/api/templates/[id]/route.test.ts`**: Validating individual modification rules like isolating specific layout records targeting deletion edge cases safely.

## How the architecture comes together

When a user visits **PrintGenerator**, the flow of execution spans across all these interconnected layers seamlessly:

1. **The User Interface (`app/` & `app/components/`)**: The user interacts with the React pages starting at `app/page.tsx`, which organizes the visual layouts. By interacting with the editor modals (`TemplateEditor.tsx`, `CreateTemplateModal.tsx`), users define designs and place dynamic variables using our augmented `tiptap` logic.
2. **The Routing Layer (`app/api/`)**: When a user clicks "Generate" or "Save", the React frontend fires HTTP requests to our Next.js backend API routes. These endpoints (like `api/templates/[id]/route.ts`) act as gateways, validating requests and managing client-server communication loops.
3. **Database Synchronization (`lib/mongodb.ts` & `types/`)**: Valid commands are passed down to our MongoDB connection handler which executes Type-safe operations securely against the database, relying heavily on document schemas defined within `types/template.ts`.
4. **The Compilation Engine (`lib/document-generation.ts`)**: For the core feature - batch generating PDFs - the API defers the heavy lifting to the document generation utility. Here, stored structural data is merged with the user's uploaded JSON dataset. The engine translates the populated object tree back into visual HTML representations and mounts Headless Chrome (via Puppeteer) locally to print physical PDF byte arrays. These are then compressed and returned as a downloadable ZIP package back up through the hierarchy to the requesting user!
